# C4 需求文档

> **版本**：v0.3.0 | **最后更新**：2026-07-11 | **来源文档**：`c4_description.md` v0.2.1

## 1. 概述与边界

| 编号 | 需求描述 | 来源 |
|------|---------|------|
| C4_RS_00001 | C4 应能自动完成数据接入流程中的意图理解、方案规划、协议配置、异常诊断等环节，替代人工逐一配置的方式 | §1 |
| C4_RS_00002 | C4 的 AI 智能体不在实时数据路径中运行——AI 负责理解、规划、配置和监控，不参与数据搬运 | §1 |
| C4_RS_00003 | 确定性的协议级数据搬运应由 MCP 服务负责，AI 不得干涉 | §1 |
| C4_RS_00004 | C4 不是 SCADA 系统、数据历史库、安全网关或 BI 平台 | §1.1 |
| C4_RS_00005 | C4 应能被不具备计算机专业知识的场站工作人员使用，通过自然语言和任意格式的文档即可完成数据接入 | §1 |

## 2. 部署与运行环境

| 编号 | 需求描述 | 来源 |
|------|---------|------|
| C4_RS_00010 | C4 应能运行于工业现场环境并与工业设备通信协议（如 Modbus、IEC 系列）交互，同时应能对接 IT 系统（如数据库写入、数据转发） | §2.1 |
| C4_RS_00011 | C4 应支持多层子网部署，包括 I 区、II 区、III 区和中心侧 | §2.2/§2.3 |
| C4_RS_00012 | C4 应支持在有网络隔离的多层子网环境中运行 | §9.1 |
| C4_RS_00013 | C4 应部署在工业现场的数据服务器上 | §2.3 |
| C4_RS_00014 | C4 应以守护进程模式持续运行，在资源可用的情况下不因自身原因退出或停止响应 | §9.1 |

## 3. 系统架构

### 3.1 Agent 与 MCP 体系

| 编号 | 需求描述 | 来源 |
|------|---------|------|
| C4_RS_00020 | C4 每个实例应由一个 Agent 和一个 MCP 服务集群组成 | §3.1 |
| C4_RS_00021 | Agent 应具备基于 LLM 的意图理解和任务推理能力 | §3.1 |
| C4_RS_00022 | Agent 与 MCP 服务之间的通信应基于 Model Context Protocol 标准 | §3.1 |
| C4_RS_00023 | MCP 服务应封装确定性的数据接入能力，对外暴露标准化的 MCP 接口（tools、resources） | §5.1 |
| C4_RS_00024 | 新协议的数据接入支持应通过新增 MCP 服务实现，遵循标准 MCP 接口 | §3.1 |
| C4_RS_00025 | MCP 服务应支持按需启动和停止，由 Agent 管理完整生命周期 | §5.4 |

### 3.2 设计原则

| 编号 | 需求描述 | 来源 |
|------|---------|------|
| C4_RS_00030 | Agent 故障不应影响已启动并正在运行的 MCP 数据管道 | §3.2 |
| C4_RS_00031 | MCP 服务在 Agent 不可用期间应继续自主运行，不中断数据采集（C4_RS_00030 的 MCP 侧实现） | §5.2 |
| C4_RS_00032 | Agent 恢复后应能自动接续对 MCP 服务的监控状态 | §3.2 |
| C4_RS_00033 | Agent 应能对常规操作（如启动已知协议的 MCP、标准监控、已知故障恢复）自主执行，无需每次人工介入 | §3.2 |
| C4_RS_00034 | MCP 服务应保证确定性——同一配置和输入下始终产生相同的采集和转发结果 | §3.2/§5.2 |

## 4. Agent 能力

### 4.1 用户输入

| 编号 | 需求描述 | 来源 |
|------|---------|------|
| C4_RS_00040 | Agent 应接收并响应用户的自然语言指令 | §4.1 |
| C4_RS_00041 | Agent 应接收并解析结构化文档（Excel、CSV 等） | §4.1 |
| C4_RS_00042 | Agent 应接收并解析非结构化文档（PDF、Word、图片、截图等） | §4.1 |
| C4_RS_00043 | Agent 应支持上述输入方式的任意混合组合 | §4.1 |
| C4_RS_00044 | Agent 应从输入中自动提取关键信息——包括设备地址、寄存器映射、数据类型、通信参数等 | §4.1 |
| C4_RS_00045 | C4 应通过 Web 界面与用户交互，支持在浏览器中提交需求、上传文档、查看运行状态和诊断报告 | §4.1 |

### 4.2 任务处理

| 编号 | 需求描述 | 来源 |
|------|---------|------|
| C4_RS_00050 | Agent 应能从用户输入中提取数据接入意图，并产出可执行的数据接入方案（包含通信协议、目标设备、数据点清单、转发目标），供用户确认 | §4.2 |
| C4_RS_00051 | Agent 应将接入需求分解为可执行的步骤——包括选择通信协议、确定配置参数、确定需要启动的 MCP 服务 | §4.2 |
| C4_RS_00052 | Agent 应能启动、配置和停止 MCP 服务 | §4.2 |
| C4_RS_00053 | Agent 在启动 MCP 后应自动完成协议级参数配置 | §7.1 |

### 4.3 监控与异常处理

| 编号 | 需求描述 | 来源 |
|------|---------|------|
| C4_RS_00060 | Agent 应持续监控各 MCP 服务的运行状态 | §4.2 |
| C4_RS_00061 | Agent 应监控 MCP 服务的数据采集指标——至少包括采集成功率、数据时效性、数据完整性——并在指标偏离正常范围时触发异常处理流程 | §4.2 |
| C4_RS_00062 | Agent 应在发现异常时分析根因，并生成包含异常现象描述、可能原因列表、建议操作步骤的诊断报告 | §4.2 |
| C4_RS_00063 | Agent 应采用分级错误处理策略：自动恢复 / 诊断上报 / 人工审核 | §4.3 |
| C4_RS_00064 | 对可自动处理的故障（如通信连接中断），Agent 应执行自动修复，无需人工介入 | §4.3 |
| C4_RS_00065 | 对需要领域知识判断的异常（如数据质量问题），Agent 应生成诊断报告并建议解决方案 | §4.3 |
| C4_RS_00066 | 涉及数据接入配置的任何变更，Agent 应在人工审核确认后方可生效（与 C4_RS_00121 交叉引用） | §4.3 |
| C4_RS_00067 | Agent 在新数据源接入确认正常后应通知用户 | §7.1 |
| C4_RS_00068 | C4 应支持对达到设定等级的故障通过邮件或其他渠道向指定人员发送告警通知，故障等级和通知渠道可配置 | §4.3 |

## 5. MCP 服务

### 5.1 通用职责

| 编号 | 需求描述 | 来源 |
|------|---------|------|
| C4_RS_00080 | MCP 服务应接收 Agent 的配置指令，完成协议级参数配置 | §5.2 |
| C4_RS_00081 | MCP 服务应执行确定性的数据采集工作 | §5.2 |
| C4_RS_00082 | MCP 服务应执行确定性的数据转发工作 | §5.2 |
| C4_RS_00083 | MCP 服务应向 Agent 报告运行状态、数据统计和异常信息 | §5.2 |
| C4_RS_00084 | MCP 服务应以守护进程模式持续运行，在 Agent 不可用期间继续执行数据采集和转发，不因 Agent 故障而中断 | §5.2 |
| C4_RS_00085 | MCP 服务应执行数据转换工作——如协议格式转换、数值单位换算、数据结构重组 | §5.2 |

### 5.2 协议支持

| 编号 | 需求描述 | 来源 |
|------|---------|------|
| C4_RS_00090 | C4 应支持通过 Modbus TCP/RTU 协议采集设备寄存器数据 | §5.3 |
| C4_RS_00091 | C4 应支持通过 IEC 60870-5-104 协议采集远动数据 | §5.3 |
| C4_RS_00092 | C4 应支持通过 IEC 60870-5-101 协议采集串口远动数据 | §5.3 |
| C4_RS_00093 | C4 应支持将采集到的数据按指定格式转发给第三方系统 | §5.3 |
| C4_RS_00094 | C4 应支持将采集到的数据写入指定的数据库 | §5.3 |
| C4_RS_00095 | C4 应支持通过 ASFP2 协议接收和发送数据 | §5.3 |

## 6. C4 网络

| 编号 | 需求描述 | 来源 |
|------|---------|------|
| C4_RS_00100 | 不同子网（I 区、II 区、III 区、中心侧）的 C4 Agent 应能构成统一的 C4 网络 | §6.1 |
| C4_RS_00101 | C4 Agent 之间应支持直接对等通信，不依赖统一管理节点 | §6.1 |
| C4_RS_00102 | Agent 之间的通信应仅交换任务协调信息和数据流元数据，不得传输原始采集数据 | §6.1 |
| C4_RS_00103 | 原始采集数据的传输应由各子网的 MCP 服务按配置的数据流向执行 | §6.1 |
| C4_RS_00104 | C4 网络应保证每条数据从生产端到消费端的路径可追溯——可确定数据的产生设备、经过的子网节点和到达中心侧的时间 | §6.2 |

## 7. 安全

### 7.1 安全能力

| 编号 | 需求描述 | 来源 |
|------|---------|------|
| C4_RS_00110 | C4 应验证访问 Agent 的用户身份 | §8.1 |
| C4_RS_00111 | C4 应支持 Agent 实例间的互认身份 | §8.1 |
| C4_RS_00112 | C4 应支持基于角色的操作授权——关键配置变更须经授权审核 | §8.1 |
| C4_RS_00113 | C4 应对跨安全分区（I/II/III 区 / 中心侧）的通信链路进行加密 | §8.1 |
| C4_RS_00114 | C4 应记录关键操作——包括配置修改、MCP 启停、数据转发变更——以支持事后审计 | §8.1 |

### 7.2 安全原则

| 编号 | 需求描述 | 来源 |
|------|---------|------|
| C4_RS_00120 | MCP 服务应仅拥有执行其数据接入任务所需的最小权限 | §8.2 |
| C4_RS_00121 | 任何涉及数据接入配置的变更应在人工审核确认后方可生效（与 C4_RS_00066 同义，从安全视角重申） | §8.2 |
| C4_RS_00122 | 跨子网通信应符合场站网络安全分区管理规定 | §8.2 |
| C4_RS_00123 | C4 自身不替代场站已有的安全基础设施（防火墙、网闸、入侵检测等），应在已有安全边界内运行 | §8.3/§1.1 |

## 8. 数据治理

| 编号 | 需求描述 | 来源 |
|------|---------|------|
| C4_RS_00130 | 用户上传的配置文档和数据所有权应归属于场站方 | §2.4 |
| C4_RS_00131 | C4 不应持久化存储采集的数据内容——数据应经 MCP 管道流式转发，不在 C4 内部落地 | §2.4 |

## 10. 许可管理
